# 阶段产出物契约

每个功能开发必须产出以下文件，确保可追溯、可审查。

## 产出物总览

| 阶段 | 产出物 | 文件名 | 必须 |
|------|--------|--------|------|
| 方案编写 | 方案文档 | `spec.md` | 是 |
| 方案审查 | 审查报告 | `spec-review.md` | 是 |
| 方案修订 | 更新的方案 | `spec.md`（更新） | 是 |
| 代码实现 | 功能源码 | 按命名规范 | 是 |
| 代码实现 | 单元测试 | `*.test.ts` | 建议 |
| 代码审查 | 审查报告 | `code-review.md` | 是 |
| 代码修订 | 修订后源码 | 更新的源码 | 是 |
| 验收 | 验收报告 | `acceptance.md` | 是 |

## spec.md — 方案文档

产生于方案编写阶段，可能在修订阶段更新。

必须包含的章节：

| 章节 | 内容 |
|------|------|
| 需求理解 | 需求重述、范围和边界 |
| 架构落点 | 涉及的层级、API、Entity/Display/Command/Canvas 分析 |
| 接口设计 | 类签名、注册方式、继承关系（TypeScript 签名） |
| 数据流分析 | 用户操作到渲染更新的完整路径（Mermaid flowchart） |
| 风险与陷阱 | 至少列出 3 个相关风险 |
| 验收标准 | 可测试的验收条件，每条可明确判定通过/不通过 |
| 任务拆分 | 实现步骤，粒度到文件级，标注依赖关系 |

每次修订在文档末尾追加变更记录。

## spec-review.md — 方案审查报告

产生于方案审查阶段。

必须包含：

- **基本信息**：方案名称、审查人、日期、版本
- **评分**（6 个维度）：

| 维度 | 权重 |
|------|------|
| 架构合规性 | 25% |
| 生命周期安全 | 20% |
| API 使用正确性 | 15% |
| 模式遵循 | 15% |
| 风险覆盖 | 15% |
| 验收清晰度 | 10% |

- **通过条件**：综合 >= 7.0，无单一维度 < 5.0，架构合规性 >= 6.0，生命周期安全 >= 6.0
- **问题列表**：编号、维度、严重程度、描述、建议修改
- **结论**：通过 / 有条件通过 / 不通过

## code-review.md — 代码审查报告

产生于代码审查阶段。

必须包含：

- **审查范围**：审查的文件列表（文件名、类型、行数）
- **方案一致性**：实现是否与 spec.md 一致
- **代码质量**：
  - 架构合规：Entity 只管数据、Display 通过 Signal 响应、Command 交互封装
  - 生命周期安全：listen/unlisten 配对、dirty 调用、freeze/unfreeze 在 try/finally、onCleanup 清理 Temp 层
  - API 使用：导入路径、Display 注册顺序（子类先于父类）、命令注册、图层类型
- **问题列表**：编号、文件、行号、严重程度（高/中/低）、描述、修复建议
- **结论**：通过 / 有条件通过 / 不通过

## acceptance.md — 验收报告

产生于验收阶段，是功能开发的最终产出。

必须包含：

- **功能验收**：逐条验证 spec.md 中的验收标准
- **质量验收**：Signal 泄漏、内存泄漏、脏标记、注册顺序、命名规范、资源销毁
- **性能验收**：帧率、内存占用、响应延迟（需实际测量数据）
- **遗留问题**：编号、描述、严重程度、后续计划
- **结论**：验收通过 / 验收不通过

## 文件组织

```
features/
└── <功能名>/
    ├── spec.md              # 方案（编写 + 修订）
    ├── spec-review.md       # 方案审查报告
    ├── code-review.md       # 代码审查报告
    └── acceptance.md        # 验收报告
```

## 流转规则

- 每个产出物有且只有一个负责人，撰写者、审查者、验收者不能是同一人
- 产出物不可跳过
- spec.md 每次修订版本号 +0.1
- 审查报告归档保留，不删除历史版本
- acceptance.md 标志功能开发完成
