# 代码放置决策

## 目标

帮助业务开发时快速判断：应该在业务库扩展，还是已经超出了业务库边界，需要改底座。

## 优先放在业务库的内容

- 行业或项目特有的 `Entity`
- 行业或项目特有的 `Display`
- 业务命令、交互规则、预览对象
- 业务 `Loader`、默认场景、配置组合
- 通过 `install(app)` 组织的注册模块

## 优先留在底座的内容

- 通用的 document、selection、signal、input、canvas 生命周期
- 通用 layer / pick / dirty / displayMap 机制
- 对所有业务库都有价值的框架能力
- 需要多个业务库共享且语义稳定的基础扩展点

## 需要暂停确认的信号

- 你准备复制 `FSApp.App`、`ThreeCanvas`、`Selection`、`Document`
- 你为了一个业务需求，打算在业务库里重写通用输入或渲染流程
- 你发现多个业务库都要补同一个“缺失的扩展点”
- 需求无法通过注册、组合、继承、模块安装完成，只能改底座内部流程

## 默认判断顺序

1. 先问：现有 `CadApp` / `Cad3DCanvas` / `Loader` / `Command` 扩展点能否承载
2. 再问：能否通过注册链、模块化安装、配置组合完成
3. 最后才问：是否真的需要修改底座

## 对 reviewer 的要求

- 看到“重写底座类”时，先质疑是否误判了边界
- 看到“直接塞 `THREE.Object3D` 到 scene”时，先检查是否绕开了 document/display 链
- 看到“复制旧 `.webcad` 里的实现”时，先回源码核对当前底座是否已经变化
